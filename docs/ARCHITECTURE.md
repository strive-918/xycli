# Architecture: XYCLI

> Generated by Planner Agent | Stage 2 of software-development workflow

## 1. System Overview

XYCLI is a terminal-native AI coding agent distributed as an npm package and implemented as a TypeScript monorepo. It runs entirely from the developer's shell, turns natural-language goals into auditable coding actions, and coordinates LLM reasoning, repository context, local tools, MCP tools, plugins, session persistence, and approval gates inside one resumable CLI session.

The system is not a web service. The primary runtime boundary is the local process launched by `xycli`, with explicit integration points for LLM providers, the local filesystem, the user's shell, optional network fetches, MCP servers, and plugin entrypoints. All state needed to resume or audit a task is persisted locally in SQLite, with a JSONL crash-recovery fallback.

## 3. Component Design

### 3.1 CLI Components

| Component | Responsibility | Package |
|---|---|---|
| CLI Entry | Provides the `xycli` executable, parses argv, initializes process signals, and delegates commands. | `@xycli/cli` |
| Command Router | Routes `run`, `resume`, `config`, `session`, `plugin`, `doctor`, `diff`, `undo`, and `version` commands. | `@xycli/cli` |
| CLI Shell Renderer | Streams assistant text, tool progress, command output, diffs, approvals, and status indicators to ANSI terminals. | `@xycli/cli` |
| Agent Core | Owns the observe-plan-act-reflect loop, planning state, tool-call normalization, interruption handling, retries, and completion criteria. | `@xycli/core` |
| Tool Registry | Registers built-in tools, MCP tools, and plugin tools behind one validated execution contract. | `@xycli/tools` |
| Provider Adapter | Normalizes Anthropic, OpenAI, and OpenAI-compatible APIs into one streaming chat and tool-calling interface. | `@xycli/providers` |
| Session Manager | Persists messages, tool calls, approvals, file changes, token usage, and resumable agent state. | `@xycli/session` |
| Config Manager | Loads and validates defaults, user config, project config, environment variables, and CLI flags. | `@xycli/core` |
| Plugin Registry | Loads plugin manifests, validates entrypoints, registers plugin tools and commands, and tracks enablement state. | `@xycli/plugins` |
| Permission Engine | Applies permission levels, allowlists, denylists, secret redaction, and approval prompts before tool execution. | `@xycli/core` |
| Context Manager | Selects, ranks, excerpts, estimates, compacts, and assembles model context before each provider request. | `@xycli/core` |

### 3.2 CLI Command Layer

- **Protocol:** Local process invocation through Node.js CLI.
- **Executable:** `xycli`.
- **Command Surface:** `run`, `resume`, `config`, `session`, `plugin`, `doctor`, `diff`, `undo`, `version`.
- **Authentication:** No XYCLI login. Provider credentials are read from configured environment variables or OS keychain references.
- **Exit Codes:** `0` success, `1` general error, `2` validation error, `3` permission denied, `4` provider error, `5` tool error.

### 3.3 Agent Layer

- **Pattern:** Local orchestrator with deterministic state transitions and explicit side-effect boundaries.
- **Key Domain Objects:**
  - `AgentLoop`
  - `Plan`
  - `Observation`
  - `ToolCall`
  - `ToolResult`
  - `ProviderMessage`
  - `Session`
  - `ApprovalRequest`
  - `PermissionPolicy`
  - `ContextBundle`

### 3.4 Data Layer

- **Primary Store:** SQLite database at `.xycli/sessions/xycli.sqlite3` by default.
- **Fallback Store:** Append-only JSONL files at `.xycli/sessions/recovery/*.jsonl` for crash recovery when SQLite is temporarily unavailable.
- **Cache:** In-process LRU caches for config resolution, file metadata, token estimates, and provider capability discovery.
- **Queue:** In-process serialized tool scheduler. XYCLI does not use an external queue in v1.

## 4. Tool System Architecture

### 4.1 Tool Interface

Every tool is registered through one exact contract:

```ts
export type PermissionLevel =
  | "read-only"
  | "write-files"
  | "run-safe-commands"
  | "network"
  | "full-access";

export interface ToolExecutionContext {
  sessionId: string;
  callId: string;
  cwd: string;
  env: Record<string, string>;
  signal: AbortSignal;
  permissions: PermissionPolicy;
  logger: StructuredLogger;
}

export interface ITool<TInput extends object = object, TOutput = unknown> {
  name: string;
  description: string;
  inputSchema: JSONSchema7;
  permissionLevel: PermissionLevel;
  defaultTimeoutMs: number;
  idempotencyKey(input: TInput, context: ToolExecutionContext): string;
  execute(input: TInput, context: ToolExecutionContext): Promise<ToolResult<TOutput>>;
}

export interface ToolResult<TOutput = unknown> {
  success: boolean;
  output: TOutput | null;
  error: ToolErrorPayload | null;
  durationMs: number;
  startedAt: string;
  endedAt: string;
  metadata: Record<string, unknown>;
}
```

### 4.2 Built-In Tools

| Tool | Permission | Purpose |
|---|---|---|
| `file_read` | `read-only` | Read a whole file or byte/line range up to configured size limits. |
| `file_write` | `write-files` | Create or replace a file through a diff-backed write with pre/post hashes. |
| `file_patch` | `write-files` | Apply unified patches with validation, preview, and rollback metadata. |
| `file_delete` | `full-access` | Delete a file only after high-risk approval. |
| `directory_list` | `read-only` | List directory entries with ignore rules and size metadata. |
| `search_text` | `read-only` | Run `rg`-compatible local text search with result limits and line excerpts. |
| `terminal_exec` | `run-safe-commands` | Execute shell commands with streaming output, timeout, risk scanning, and exit-code capture. |
| `web_fetch` | `network` | Fetch and excerpt public web pages under network policy and size limits. |
| `git_status` | `read-only` | Inspect repository state, branch, HEAD, dirty files, and staged changes. |
| `git_diff` | `read-only` | Show working tree, staged, or session-scoped diffs. |

### 4.3 MCP Bridge

XYCLI acts as an MCP client. Each configured MCP server is started or connected during initialization, discovered tools are converted into `ITool` instances, and all MCP tool calls pass through the same schema validation, permission policy, timeout, audit logging, and session persistence as built-in tools.

Supported MCP transport in v1 is stdio. Remote MCP over HTTP is represented as a network tool and requires the `network` permission level.

### 4.4 Plugin Tools

Plugins are local npm-compatible packages with a manifest. The plugin registry loads the manifest, validates the declared permissions, imports the entrypoint in an isolated module context, and registers tools through the same `ToolRegistry.register(tool)` path. Plugin tools cannot directly receive privileged process objects; all filesystem, command, and network effects must go through declared tool execution APIs and approval gates.

## 5. Agent Loop

The agent loop is a resumable state machine:

```
observe → plan → act → reflect
   ▲                    │
   └────────────────────┘
```

1. **Observe:** Load user goal, session history, custom instructions, repository state, recent tool results, and relevant file excerpts.
2. **Plan:** Ask the provider for the next assistant response and normalized tool calls. The context manager compacts before the request exceeds 90% of the model window.
3. **Act:** Validate tool inputs, evaluate permissions, request approval when needed, execute tools with timeouts, and stream progress.
4. **Reflect:** Persist observations, summarize results, update the active plan, decide whether to continue, ask the user, or complete.

Streaming is first-class. Provider tokens, tool progress, command stdout/stderr, status indicators, approval prompts, and final summaries are emitted through one event stream consumed by the CLI renderer.

Interrupt handling is deterministic. On `Ctrl+C`, the command router sends an abort signal to the active loop, the tool scheduler stops dispatching new tools, the running tool receives `AbortSignal`, and the session manager must flush current state within two seconds.

Approval flow is enforced before side effects. The permission engine classifies each action, applies allowlists and denylists, redacts secrets, and returns one of `allow`, `deny`, or `prompt`. High-risk operations always display action, target, reason, and potential impact.

## 6. Provider Layer

The provider layer has one normalized interface:

```ts
export interface IProvider {
  name: "anthropic" | "openai" | "generic-openai";
  chat(request: ProviderRequest): Promise<ProviderResponse>;
  streamChat(request: ProviderRequest): AsyncIterable<ProviderStreamEvent>;
  supportsTools(model: string): boolean;
  estimateTokens(input: ProviderTokenInput): Promise<TokenEstimate>;
}
```

XYCLI ships three adapters:

| Adapter | Purpose |
|---|---|
| `AnthropicProvider` | Calls Anthropic Messages API and maps provider tool_use/tool_result events to internal `ToolCall`. |
| `OpenAIProvider` | Calls OpenAI Responses API and maps function/tool events to internal `ToolCall`. |
| `GenericOpenAIProvider` | Calls OpenAI-compatible chat APIs using configured `baseUrl`, model, headers, and tool schema support. |

Provider requests use retry with exponential backoff and jitter for transient `429`, `500`, `502`, `503`, and network reset errors. After two consecutive transient failures on the primary provider, the circuit breaker opens for the configured cool-down period and the session switches to the configured fallback provider. Tool-calling is normalized before it reaches the agent loop, so downstream planning and auditing never depend on provider-specific wire formats.

## 7. Session & Persistence

SQLite is the authoritative store. Each session records user messages, assistant messages, provider metadata, tool calls, approvals, file changes, diffs, token usage, context compactions, and final status. Writes are transactional around each loop iteration so a crash does not leave partial tool records without their related observation.

The JSONL fallback is used only when SQLite cannot be opened or when crash recovery needs an append-only emergency log. On the next successful startup, `xycli doctor` and `xycli session recover` import JSONL records back into SQLite.

High-level schema:

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    cwd TEXT NOT NULL,
    status TEXT NOT NULL,
    provider_name TEXT NOT NULL,
    model TEXT NOT NULL,
    fallback_provider_name TEXT,
    fallback_model TEXT,
    current_state TEXT NOT NULL,
    plan_json TEXT NOT NULL DEFAULT '{}',
    git_branch TEXT,
    git_head TEXT,
    dirty_files_count INTEGER NOT NULL DEFAULT 0,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    completed_at TEXT
);
```

Complete DDL is defined in `docs/DESIGN.md`.

## 8. Configuration

Configuration is resolved through exactly four user-controlled layers:

```
CLI flag > environment variable > project .xycli/config.yaml > user ~/.xycli/config.yaml > built-in defaults
```

The config manager validates the merged result before any agent loop starts. Invalid fields produce path-specific errors and exit code `2`. Secret values are never stored directly in project config; provider keys are referenced through `apiKeyEnv`, and log output redacts values matching configured secret patterns.

The resolved config controls providers, fallback provider, tools, permissions, MCP servers, plugins, context window, instructions, telemetry, session path, shell behavior, and diagnostics.

## 10. Security Architecture

- **Authentication:** Provider credentials are read from environment variables or OS keychain references. XYCLI itself has no remote identity in v1.
- **Authorization:** Local permission policy with permission levels: `read-only`, `write-files`, `run-safe-commands`, `network`, `full-access`.
- **Approval Gate:** Every side-effecting action is evaluated as `allow`, `deny`, or `prompt`. High-risk destructive actions require explicit confirmation.
- **Allowlists/Denylist:** Policies apply to command patterns, file paths, domains, tool names, MCP servers, and plugin names.
- **Secret Redaction:** Built-in and user-defined patterns redact API keys, private keys, tokens, passwords, and credential dumps from output, logs, telemetry, and session excerpts.
- **Workspace Protection:** Dirty git files are detected before mutation, write operations store pre-image and post-image hashes, and undo refuses to overwrite user changes when hashes diverge.
- **Network Control:** `network: disabled` blocks `web_fetch`, remote MCP, telemetry, and provider calls except when an explicit provider invocation is required for the active command.

## 11. Deployment Architecture

- **Distribution:** npm package with npm workspaces and a single `xycli` bin entry.
- **Runtime:** Node.js >= 18, tested on Node.js 18, 20, and 22.
- **Platforms:** macOS 13+, Ubuntu 22.04+, Windows 11 with PowerShell or WSL.
- **Packaging:** Published package excludes tests, fixtures, local sessions, and large generated assets. Package size target is under 20 MB unpacked.
- **Execution Model:** One local CLI process per command. Long-running work is persisted as a resumable session instead of a background service.
- **CI/CD:** GitHub Actions runs lint, typecheck, unit tests, integration tests, e2e CLI tests, package-size checks, and provenance-enabled npm publishing.

## 12. Observability

- **Structured Logging:** Local JSON logs with event type, session id, tool call id, provider request id, duration, exit code, and redacted payload summaries.
- **Token Tracking:** Per-request input, output, cache read, cache write, total tokens, model, provider, and estimated cost.
- **Error Telemetry:** Opt-in only. Payloads include error class, sanitized stack fingerprint, platform, version, command, and timing metadata. Source code, prompt text, stdout/stderr bodies, and secrets are excluded.
- **Diagnostics:** `xycli doctor` checks Node version, executable path, config validity, provider credentials, provider connectivity, MCP server startup, plugin manifests, session store access, git availability, shell availability, and write permissions.
- **Local Reports:** Diagnostics can be exported as redacted JSON under 1 MB by default.

## 13. Risks & Mitigations

| Risk | Impact | Likelihood | Mitigation |
|---|---:|---:|---|
| Token cost explosion from large repositories and long sessions | High | High | Mandatory token estimation, 90% context compaction threshold, excerpt selection, duplicate-content suppression, per-session token budgets, and cost warnings. |
| LLM hallucination causes wrong edits or unsafe commands | High | Medium | Tool schema validation, explicit planning state, permission gates, diff previews, dirty-file warnings, test reruns, and auditable session logs. |
| Tool crash or process interruption loses work | High | Medium | Transactional SQLite writes per loop iteration, append-only JSONL fallback, abort signal propagation, and resume support. |
| Large repository search and context selection become slow | Medium | High | Prefer `rg`, ignore files from gitignore and config, cap search results, maintain file metadata cache, and inject excerpts instead of full files. |
| Provider outage blocks agent usage | Medium | Medium | Retry transient failures, circuit breaker after two failures, configured fallback provider, and local-only commands that work without network. |
| Plugin or MCP tool bypasses safety expectations | High | Medium | Manifest validation, declared permission levels, centralized tool execution context, approval gate enforcement, and audit logging for all external tools. |
| Secret leakage through logs or telemetry | High | Medium | Redaction before rendering, persistence, and telemetry; telemetry off by default; project config stores key env names instead of secret values. |
