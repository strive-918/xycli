# PRD: XYCLI

> Generated by Planner Agent | Stage 1 of software-development workflow

## 1. Overview

**Feature:** XYCLI 是一个 terminal-native AI coding agent，通过自然语言指令、LLM 推理和可审计 tools，在本地 CLI 工作流中完成代码编写、调试、重构、测试和部署任务。

**Background:** 开发者在日常软件工程中需要频繁切换 IDE、terminal、文档、搜索引擎和 CI/CD 系统。现有 AI coding 工具通常受限于封闭生态或图形界面，难以融入脚本化、远程服务器、SSH、CI runner 和高级自动化场景。XYCLI 目标是提供类似 Claude Code 的 terminal-first 体验，同时保持 open、extensible、model-agnostic，使团队可以接入 Claude API、OpenAI API、本地 tools、MCP servers 和自定义 plugins。

**Scope:** v1 覆盖交互式 CLI agent loop、文件读写、terminal command 执行、代码搜索、web fetch、多模型配置、session 管理、MCP protocol、自定义 tools/plugins、权限与安全控制、git 集成、context window 管理、streaming output、undo/rollback、telemetry 和 npm 分发。v1 不包含 VS Code extension、web UI、多人协作和集中式团队管理功能。

## 3. Functional Requirements

### FR1: Agent Loop
- **Description:** XYCLI 必须实现可中断、可恢复的 agent loop：理解用户目标、收集上下文、制定计划、选择 tool、执行 tool、观察结果、更新计划、输出结论，直到任务完成、失败或等待用户确认。
- **Acceptance Criteria:**
  - [ ] 对一个包含读文件、改文件、运行测试的任务，agent loop 可在单个 session 中完成至少 5 次 observe-act 迭代。
  - [ ] 每次 tool 执行后必须记录 tool name、input、exit status、stdout/stderr 摘要、耗时和时间戳。
  - [ ] 用户输入 `Ctrl+C` 时，当前 session 必须在 2 秒内停止新 tool 调度并保存状态。
  - [ ] agent 在无法继续时必须输出明确 blocker、已尝试步骤和下一步建议。
- **Dependencies:** FR2, FR8, FR14

### FR2: Tool System
- **Description:** XYCLI 必须提供统一 tool interface，支持 schema validation、权限检查、timeout、streamed output、structured result 和 audit log。
- **Acceptance Criteria:**
  - [ ] 每个 tool 必须声明 `name`、`description`、JSON schema input、permission level、default timeout 和 result schema。
  - [ ] 无效 tool input 必须在执行前被拦截，并返回可读 validation error。
  - [ ] 单个 tool 默认 timeout 为 120 秒，用户可在 config 中设置 10 秒到 3600 秒。
  - [ ] tool 执行日志必须可通过 `xycli session show <session-id>` 查看。
- **Dependencies:** FR13, FR14

### FR3: File Operations
- **Description:** XYCLI 必须支持安全的文件读取、目录列举、文本搜索、补丁写入、文件创建和文件删除，并对写操作提供 diff 预览。
- **Acceptance Criteria:**
  - [ ] 支持读取单文件最大 2 MB；超过限制时必须提示使用 range read 或 search。
  - [ ] 支持 `rg` 风格全文搜索，搜索结果默认限制 200 条并显示文件路径、行号和匹配摘要。
  - [ ] 所有写入必须生成 unified diff，且在 non-auto mode 下需要用户确认。
  - [ ] 删除文件必须被标记为 high-risk action，并要求显式确认。
- **Dependencies:** FR2, FR14, FR15

### FR4: Terminal Command Execution
- **Description:** XYCLI 必须能执行 shell commands，用于测试、构建、lint、package scripts、git 和 deployment commands，并提供 sandbox-aware 风险提示。
- **Acceptance Criteria:**
  - [ ] command 执行必须 streamed 输出 stdout/stderr，首个输出 chunk 在 command 产生内容后 200 ms 内显示。
  - [ ] 默认禁止执行包含 `rm -rf /`、`sudo`、credential dump、fork bomb 模式的 commands，除非 config 显式 allow。
  - [ ] interactive command 超过 5 秒无输出时，必须提示用户是否继续等待、发送输入或终止。
  - [ ] command result 必须包含 exit code、duration、truncated 标记和最后 200 行输出摘要。
- **Dependencies:** FR2, FR14

### FR5: Search And Web Fetch
- **Description:** XYCLI 必须支持 repo-local search 和可选 web fetch，用于读取公开文档、issue、API reference 和错误信息页面；web 能力必须可禁用。
- **Acceptance Criteria:**
  - [ ] local search 在 100k files repo 中对普通文本查询 p95 响应时间必须小于 2 秒，前提是 `rg` 可用。
  - [ ] web fetch 必须默认限制单页正文提取 100 KB，超过后摘要或分页读取。
  - [ ] web fetch 必须记录 URL、HTTP status、content type、fetch time 和使用的 excerpt。
  - [ ] 当 config 设置 `network: disabled` 时，任何 web fetch 必须被拒绝并解释原因。
- **Dependencies:** FR2, FR13, FR14

### FR6: Multi-Model Support
- **Description:** XYCLI 必须支持 Claude API、OpenAI API 和兼容 OpenAI Chat Completions/Responses 风格的 provider，并允许 primary/fallback model 配置。
- **Acceptance Criteria:**
  - [ ] 用户可通过 config 指定 provider、model、base URL、API key env var、temperature、max output tokens。
  - [ ] primary model 连续 2 次返回 rate limit 或 5xx 时，必须自动切换 fallback model 并在 session log 中记录。
  - [ ] 不同 provider 的 tool calling 必须映射到统一 internal tool call representation。
  - [ ] 缺失 API key 时，CLI 必须在启动后 1 秒内给出 provider-specific 设置提示。
- **Dependencies:** FR2, FR13

### FR7: Streaming Output
- **Description:** XYCLI 必须以 streaming 方式显示 LLM reasoning summary、assistant response、tool progress 和 command output，避免长时间无反馈。
- **Acceptance Criteria:**
  - [ ] LLM streaming time-to-first-token 在 provider 已响应情况下必须小于 500 ms。
  - [ ] 超过 3 秒没有新输出时，CLI 必须显示当前状态，例如 “thinking”、“running tests” 或 “waiting for approval”。
  - [ ] streaming 输出必须支持 ANSI terminal，且在 80 列宽下不破坏表格和代码块可读性。
  - [ ] 用户可通过 `--no-stream` 禁用 streaming，并获得完整 buffered 输出。
- **Dependencies:** FR1, FR4, FR6

### FR8: Session Management
- **Description:** XYCLI 必须保存每个 session 的 user messages、assistant messages、tool calls、diffs、approvals、token usage、model metadata 和 final status。
- **Acceptance Criteria:**
  - [ ] 默认 session 存储路径为项目内 `.xycli/sessions/`，可通过 config 覆盖。
  - [ ] `xycli resume <session-id>` 必须恢复最近一次 pending 或 completed session 的上下文和 plan。
  - [ ] `xycli session list` 必须显示最近 50 个 sessions 的 id、title、status、created time、updated time 和 token usage。
  - [ ] session 文件必须使用 JSONL 或 SQLite 等可迁移格式，且单个 session 损坏不得影响其他 sessions。
- **Dependencies:** FR1, FR13

### FR9: MCP Protocol Support
- **Description:** XYCLI 必须作为 MCP client 连接本地或远程 MCP servers，发现 tools、resources 和 prompts，并纳入 agent tool system。
- **Acceptance Criteria:**
  - [ ] 支持 stdio MCP server 配置，包含 command、args、env 和 working directory。
  - [ ] MCP tool 必须显示 server name、tool name、description 和 input schema。
  - [ ] MCP server 连接失败时，XYCLI 必须跳过该 server 并在启动诊断中显示错误。
  - [ ] MCP tool execution 必须经过与内置 tool 相同的 permission policy 和 audit log。
- **Dependencies:** FR2, FR13, FR14

### FR10: Custom Tools And Plugins
- **Description:** XYCLI 必须允许用户通过 plugin manifest 注册自定义 tools、commands、prompts 和 configuration defaults。
- **Acceptance Criteria:**
  - [ ] plugin manifest 必须支持 name、version、entrypoint、permissions、commands 和 tools 字段。
  - [ ] 无效 plugin 必须在加载时输出 validation error，并不影响其他 plugin 加载。
  - [ ] 用户可通过 `xycli plugin list` 查看 plugin 状态、版本、来源路径和启用状态。
  - [ ] plugin tool 不得绕过 FR14 权限系统，所有高风险操作必须走 approval。
- **Dependencies:** FR2, FR13, FR14

### FR11: Configuration System
- **Description:** XYCLI 必须支持 global、user、project 和 CLI flag 四层配置，并定义确定性的优先级。
- **Acceptance Criteria:**
  - [ ] 配置优先级必须为 CLI flag > environment variable > project config > user config > default config。
  - [ ] 支持 `xycli config get/set/list/validate`，validate 必须返回具体字段路径和错误原因。
  - [ ] 项目配置文件必须支持 JSON 或 YAML，默认路径为 `.xycli/config.yaml`。
  - [ ] config 中的 secret value 不得以明文出现在 logs、telemetry 或 error output 中。
- **Dependencies:** FR6, FR13, FR14

### FR12: Context Window Management
- **Description:** XYCLI 必须管理 LLM context window，通过 relevance ranking、file excerpts、summaries 和 compaction 控制 token 使用。
- **Acceptance Criteria:**
  - [ ] 每次 LLM request 前必须估算 input tokens，并在超过 model context limit 90% 时执行 compaction。
  - [ ] context selection 必须优先包含用户目标、active plan、最近 tool results、打开文件 diff 和 custom instructions。
  - [ ] 对超过 1000 行的文件，默认只注入相关 excerpt，并标注 omitted line ranges。
  - [ ] session token usage 必须按 input、output、cache read/write 或 provider 可用字段分别记录。
- **Dependencies:** FR1, FR6, FR8

### FR13: Custom Instructions
- **Description:** XYCLI 必须读取并合并 system-level、user-level 和 project-level custom instructions，用于编码风格、测试策略、安全规则和输出偏好。
- **Acceptance Criteria:**
  - [ ] 默认项目说明文件为 `.xycli/instructions.md`，缺失时不报错。
  - [ ] instructions 合并顺序必须可解释，并在 `xycli doctor` 中展示来源路径。
  - [ ] 单个 instructions 文件最大 64 KB，超过时必须提示拆分或忽略超出部分。
  - [ ] 用户可通过 `--ignore-instructions` 在单次运行中禁用 project-level instructions。
- **Dependencies:** FR11, FR12

### FR14: Permissions And Safety
- **Description:** XYCLI 必须提供权限模型，控制文件写入、命令执行、网络访问、secret 读取、MCP/plugin tool 和 destructive operations。
- **Acceptance Criteria:**
  - [ ] 内置 permission levels 至少包含 read-only、write-files、run-safe-commands、network、full-access。
  - [ ] high-risk action 必须显示 action、target、reason、potential impact，并要求用户输入明确确认。
  - [ ] secret patterns 包括 API key、private key、token、password；匹配内容在输出和 logs 中必须 redacted。
  - [ ] 用户可通过 allowlist/denylist 对 command、path、domain 和 tool name 进行策略控制。
- **Dependencies:** FR2, FR3, FR4, FR5, FR9, FR10, FR11

### FR15: Undo And Rollback
- **Description:** XYCLI 必须支持对 agent 写入的文件变更进行 undo/rollback，并优先使用 patch inverse 和 git 状态保护。
- **Acceptance Criteria:**
  - [ ] 每次文件写入必须记录 pre-image hash、post-image hash 和 unified diff。
  - [ ] `xycli undo <session-id>` 必须能回滚该 session 中最近一次未回滚的文件变更。
  - [ ] 如果目标文件已被用户后续修改且 hash 不匹配，undo 必须停止并展示 conflict。
  - [ ] rollback 不得删除 session 开始前已存在且未由 XYCLI 创建的用户文件。
- **Dependencies:** FR3, FR8, FR16

### FR16: Git Integration
- **Description:** XYCLI 必须理解 git workspace 状态，辅助创建分支、查看 diff、生成 commit message、解释 conflicts 和保护用户未提交改动。
- **Acceptance Criteria:**
  - [ ] session 开始时必须检测是否在 git repo 中，并记录 current branch、HEAD commit 和 dirty files 数量。
  - [ ] 在修改已 dirty 文件前，XYCLI 必须提示该文件已有未提交改动。
  - [ ] `xycli diff` 必须显示当前 session 产生的 diff 和 working tree diff 的区别。
  - [ ] `xycli commit --suggest` 必须基于 staged diff 生成不超过 72 字符 summary 和可选 body。
- **Dependencies:** FR3, FR8, FR14, FR15

### FR17: Telemetry And Diagnostics
- **Description:** XYCLI 必须提供 opt-in telemetry 和本地 diagnostics，用于性能、错误、tool usage 和 token usage 分析，同时保护代码和 secret。
- **Acceptance Criteria:**
  - [ ] telemetry 默认关闭，首次启动必须提供明确 opt-in/skip 选择。
  - [ ] telemetry payload 不得包含源码、prompt 原文、command stdout/stderr 原文或 secret。
  - [ ] `xycli doctor` 必须检查 Node.js version、API key presence、provider connectivity、MCP config、plugin config 和 write permissions。
  - [ ] diagnostics report 必须可导出为 redacted JSON，单文件大小默认小于 1 MB。
- **Dependencies:** FR6, FR9, FR10, FR11, FR14

### FR18: CLI Packaging And Commands
- **Description:** XYCLI 必须作为 npm package 分发，提供稳定 command surface、help 文档和 exit codes，支持 macOS、Linux 和 Windows terminal。
- **Acceptance Criteria:**
  - [ ] `npm install -g xycli` 后必须提供 `xycli` executable。
  - [ ] `xycli --help` 必须列出 run、resume、config、session、plugin、doctor、diff、undo、version commands。
  - [ ] 所有 commands 必须使用稳定 exit codes：0 success、1 general error、2 validation error、3 permission denied、4 provider error、5 tool error。
  - [ ] 发布包解压后 npm package size 必须小于 20 MB，不包含 tests fixture 大文件和本地 session 数据。
- **Dependencies:** FR8, FR11, FR17

## 5. Constraints

- Runtime 必须使用 Node.js >= 18。
- Implementation language 必须以 TypeScript 为主，核心包必须生成 typed declarations。
- Distribution 必须通过 npm package 提供，并支持 global install 和 project-local install。
- License 必须为 MIT license。
- v1 必须在没有网络时支持 local-only operations：文件读写、搜索、git diff、session 查看、undo、config 和本地 command 执行。
- LLM provider API key 必须通过 environment variable、OS keychain 或 user config 引用，不得要求写入项目源码。
- 默认行为必须保护用户 workspace：未确认不得执行 high-risk destructive operations。
- Core architecture 必须允许替换 model provider、tool provider 和 session store，不得硬编码单一 vendor。
- Windows 支持不得依赖 POSIX-only shell；需要兼容 PowerShell 或明确使用 WSL。

## 6. Out of Scope

- VS Code extension、JetBrains plugin 或其他 IDE extension。
- Web UI、desktop GUI 或 browser-based chat interface。
- Team collaboration features v1，例如 shared workspace、multi-user session、RBAC admin console。
- Cloud-hosted XYCLI backend、centralized policy server 或 hosted session storage。
- Autonomous long-running production deployment without user approval。
- Built-in code review SaaS、PR comment bot 或 GitHub App。
- Fine-tuning、自托管 model serving 或 vector database service。

## 7. Assumptions

- 用户是熟悉 terminal、git、package manager 和常见 build/test command 的开发者。
- 用户拥有至少一个可用 LLM API key，例如 Claude API 或 OpenAI API。
- 用户理解 agent 生成的代码需要 review，并愿意在 high-risk action 前做人工确认。
- 目标项目大多数为文本源码仓库，单 repo 文件数量不超过 100k，单文件通常小于 2 MB。
- 用户所在环境允许安装 npm package，并可运行 Node.js >= 18。
- 企业用户可通过 MCP、plugins 和 config policy 接入内部 tools，而不是要求 v1 内置所有企业系统。

## 8. Glossary

| Term | Definition |
|------|------------|
| Agent Loop | AI agent 反复执行“理解目标、选择 tool、观察结果、更新计划”的控制循环，直到任务完成或需要用户介入。 |
| Tool | 可被 agent 调用的能力单元，例如 file read、file write、terminal command、search、web fetch、MCP tool 或 plugin tool。 |
| MCP | Model Context Protocol，用于让 AI client 连接外部 tools、resources 和 prompts 的开放协议。 |
| Context Window | LLM 单次 request 可接收和生成的 token 上限，包括 system prompt、instructions、history、tool results 和代码片段。 |
| Token | LLM 处理文本的计量单位，用于计算 context 大小、成本和输出限制。 |
| System Prompt | 发送给 LLM 的最高优先级行为说明，用于定义 agent 身份、规则、安全边界和输出格式。 |
| Session | 一次 XYCLI 交互任务的持久化记录，包含 messages、tool calls、diffs、approvals、token usage 和 status。 |
| Provider | 提供 LLM API 的服务，例如 Anthropic、OpenAI 或 OpenAI-compatible endpoint。 |
| Permission Policy | 决定 tool 是否可执行、是否需要确认以及哪些 path、command、domain 被允许或拒绝的规则集合。 |
| Rollback | 基于已记录 diff 和 file hash 撤销 XYCLI 对 workspace 产生的变更。 |
